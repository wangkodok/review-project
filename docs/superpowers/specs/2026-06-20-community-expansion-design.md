# 익명 음식 리뷰 플랫폼: 커뮤니티 확장 설계

## 1. 목적과 범위

이 문서는 기존 익명 음식 리뷰 플랫폼 MVP를 유지한 채 다음 기능을 확장하기 위한 설계 기준이다. 이 문서는 구현 지시가 아니며, 승인 전에는 데이터베이스, API, UI를 변경하지 않는다.

- 카테고리 기반 게시글 분류와 필터
- 이미지 레퍼런스에 맞춘 커뮤니티 목록 정보 구조 개편
- 목록의 작성자 전용 수정·삭제 진입점
- 검색 전용 페이지와 로그인 사용자 최근 검색어

다음은 이번 확장에서 제외한다.

- Supabase Auth, Supabase RLS, 브라우저의 Supabase 직접 접근
- 별도 하트 반응, 반응 테이블, 반응 카운터
- 배지, 신뢰도 점수, 맛집 데이터, 리뷰·평점
- 검색 랭킹, 추천 검색어, 검색어 분석 로그
- 카테고리 관리자 페이지와 카테고리 CRUD UI
- 이미지 업로드, 댓글, 신고, 무한 스크롤

## 2. 유지할 MVP 원칙

- 읽기는 공개, 쓰기·수정·삭제·좋아요·최근 검색어 저장은 Google 로그인 사용자만 가능하다.
- 모든 DB 호출은 `Client -> Next.js Route Handler -> server-side service -> Supabase` 경로를 따른다.
- `SUPABASE_SERVICE_ROLE_KEY`는 서버에만 둔다.
- 기존 게시글 삭제는 Hard Delete이며, `likes`와 `post_views`는 현재의 `ON DELETE CASCADE` 관계를 유지한다.
- 공개 API는 DB의 `snake_case`를 노출하지 않고 `camelCase` 응답 DTO를 제공한다.

### 2.1 공유 DB 운영 원칙

- 현재 로컬 개발 환경과 Vercel 배포 환경은 같은 Supabase 프로젝트를 사용한다.
- 따라서 로컬에서 수행하는 작성·수정·삭제 테스트도 배포 서비스 데이터에 반영된다.
- DB를 초기화하거나 기존 테이블을 재생성하지 않는다. 모든 스키마 변경은 Git으로 버전 관리되는 migration 파일과 검토된 SQL을 통해서만 적용한다.
- migration은 `추가(Expand) -> 새 코드 배포(Migrate) -> 데이터 검증(Validate) -> 제약 강화(Enforce)` 순서를 따른다.
- 실제 외부 사용자가 유입되기 전에는 개발·운영 DB 분리, 백업, 테스트 데이터 관리 정책을 다시 검토한다.

## 3. 현재 데이터 전제

- 테스트용 기존 게시글은 모두 삭제된 상태다.
- 따라서 기존 게시글에 `미분류`를 채우는 데이터 이전은 하지 않는다.
- 새 카테고리 적용 시점부터 모든 게시글은 실제 카테고리를 하나 가진다.
- 현재 검증된 FK cascade는 다음과 같다.
  - `posts.user_id -> users.id`
  - `likes.user_id -> users.id`, `likes.post_id -> posts.id`
  - `post_views.user_id -> users.id`, `post_views.post_id -> posts.id`

## 4. 카테고리 설계

### 4.1 사용자 정책

- 초기 카테고리는 `한식`, `중식`, `일식`이다.
- `전체`는 커뮤니티 목록의 필터 값이며, 게시글에 저장하는 카테고리가 아니다.
- 새 글 작성 시 작성자는 카테고리 하나를 필수로 선택한다. 기본 선택값은 두지 않는다.
- 수정 화면에는 게시글의 현재 카테고리 선택값을 표시한다. 활성 카테고리인 경우에만 기존 선택값을 유지한 채 변경을 허용한다.
- 목록 상단의 pill은 목록 필터다. 각 게시글 제목 위의 pill은 해당 게시글의 실제 카테고리다.

### 4.2 숨김과 삭제

- 카테고리와 게시글 삭제는 별개다. 게시글을 삭제해도 카테고리는 삭제하거나 숨기지 않는다.
- 카테고리 물리 삭제는 운영 기능으로 제공하지 않는다.
- 향후 카테고리를 운영에서 중지할 때는 `is_active = false`로 숨긴다.
- 숨긴 카테고리는 새 글 작성과 목록 필터에서 제외한다. 기존 게시글은 기존 카테고리 정보를 보존하며, 필요하면 카테고리를 다시 활성화할 수 있다.
- 추천 카테고리와 카테고리 숨김은 다른 기능이다. 추천 기능은 이번 범위에 포함하지 않는다.

### 4.3 수정 화면의 예외 처리

- 숨김 카테고리 또는 카테고리 관계가 없는 글은 수정 진입 전에 `현재 카테고리는 선택할 수 없습니다. 게시글을 수정하려면 새로운 카테고리를 선택해주세요.` 팝업을 표시한다.
- 목록 또는 상세에서 수정 클릭 후 `취소`를 누르면 팝업만 닫히고 현재 화면을 유지한다. `확인`을 누르면 수정 화면으로 이동한다.
- 수정 화면에서는 기존 제목·내용을 표시하되 활성 카테고리를 새로 선택하기 전까지 저장을 막는다. 자동 분류하거나 기존 숨김 카테고리를 유지해 저장하지 않는다.
- 직접 수정 URL로 진입한 경우에도 같은 팝업을 표시하며, `취소`는 게시글 상세로 돌아간다.
- 목록·상세·수정 화면은 서버가 파생해 제공하는 `requiresCategorySelection` 값으로 이 상태를 일관되게 처리한다.
- 새 글 작성 API는 활성 `categoryId`를 반드시 요구한다.
- 수정 API는 현재 카테고리가 활성일 때만 `categoryId` 생략을 허용하고 기존 카테고리를 유지한다. `categoryId`를 전달하면 서버는 활성 카테고리인지 확인한 뒤에만 변경한다.
- 현재 카테고리가 숨김이거나 관계가 없으면 수정 API는 활성 `categoryId`를 요구하며, 자동 분류하거나 임의 카테고리를 할당하지 않는다.

### 4.4 데이터 모델

`categories` 테이블을 새로 만든다.

| 필드 | 역할 |
| --- | --- |
| `id` | UUID PK |
| `name` | 화면 표시명 (`한식`, `중식`, `일식`) |
| `slug` | API/코드에서 사용하는 고정 식별자, unique |
| `sort_order` | 필터와 작성 UI 표시 순서 |
| `is_active` | 새 글·필터 노출 여부 |
| `created_at`, `updated_at` | 감사·정렬 기준 |

`posts.category_id`를 `categories.id`로 연결한다.

- `category_id`는 `NOT NULL`이다.
- FK 삭제 규칙은 `ON DELETE RESTRICT`로 둔다. 카테고리는 물리 삭제가 아니라 숨김으로 운영한다.
- 초기 데이터는 `한식`, `중식`, `일식`을 `sort_order` 순서로 넣는다.

### 4.5 카테고리 API

- `GET /api/categories`: 활성 카테고리만 `sort_order` 순서로 반환한다.
- 카테고리 목록은 클라이언트에 하드코딩하지 않는다. 새 카테고리나 숨김 정책을 반영할 때 UI 배포 없이 서버 데이터로 바꿀 수 있기 때문이다.
- 카테고리 관리 API와 관리자 UI는 만들지 않는다.

## 5. 커뮤니티 목록 디자인

### 5.1 화면 구조

- 기존 카드형 목록을 이미지 레퍼런스와 같은 세로 행 목록형으로 바꾼다.
- 커뮤니티 목록에는 뒤로가기 버튼을 표시하지 않는다.
- 제목 위의 상단 도구 영역 오른쪽에 검색 아이콘을 둔다.
- 상단 도구 영역 왼쪽은 비워 둔다. 향후 목록 진입 구조가 달라질 때 뒤로가기를 넣을 수 있지만, 현재는 표시하지 않는다.
- `커뮤니티` 제목 아래에 `전체`, `한식`, `중식`, `일식` pill 필터를 둔다.
- 게시글 행에는 실제 카테고리 pill만 표시한다. 참고 이미지의 `맛집 데이터`, `신뢰도 100%` 등 배지성 라벨은 현재 표시하지 않는다.
- 기존 `BottomTab`의 `홈 / 커뮤니티 / 내 정보` 순서, 고정 위치, 활성 상태 처리, 아이콘은 변경하지 않는다.

### 5.2 목록 조회와 카테고리 필터

- 목록 조회는 기존 `GET /api/posts`를 유지하고, 선택적 `category` query parameter를 추가한다.
- 필터 값은 DB UUID가 아닌 카테고리 slug를 사용한다. 예: `GET /api/posts?page=1&limit=10&sort=latest&category=korean`.
- `category`가 없으면 `전체`이며, 현재 활성 카테고리 slug만 허용한다.
- 숨김 또는 존재하지 않는 slug는 `400 INVALID_CATEGORY`와 사용자 친화적 오류 메시지를 반환한다.
- 목록 UI는 `GET /api/categories`로 받은 활성 카테고리만 필터로 표시하고, 필터를 바꾸면 페이지를 1로 초기화한다.

### 5.3 게시글 행

각 행은 다음 순서를 갖는다.

1. 익명 ID와 작성일
2. 더보기 메뉴 진입점
3. 실제 카테고리 pill
4. 제목(최대 2줄)
5. 내용 미리보기(최대 2줄)
6. 좋아요 수와 조회수

공백 없는 긴 문자열도 레이아웃 밖으로 넘지 않도록 줄바꿈과 말줄임 정책을 유지한다.

- 카테고리 pill은 목록과 상세 모두 제목 바로 위 왼쪽에 둔다.
- 숨김 카테고리가 연결된 기존 글도 목록과 상세에는 원래 카테고리 이름을 표시한다.

### 5.4 글쓰기 버튼과 BottomTab

- 목록의 기존 원형 FAB는 `Plus` 아이콘과 `글쓰기` 텍스트를 포함한 `+ 글쓰기` pill 버튼으로 변경한다.
- 버튼은 BottomTab 바로 위 오른쪽에 고정하고, 이동 경로는 `/community/write`로 유지한다.
- BottomTab 자체는 변경하지 않는다.

### 5.5 통계와 아이콘

- 기존 좋아요 기능은 유지하되, 아이콘을 `Heart`에서 `ThumbsUp`으로 바꾼다.
- 표시 항목은 `좋아요(엄지)`와 `조회수(Eye)` 두 개다.
- 별도 하트 통계는 만들지 않는다. 하트 데이터·API·정책이 없는 상태에서 숫자를 표시하지 않는다.

## 6. 목록의 수정·삭제 권한

### 6.1 UI 정책

- 더보기 메뉴는 누구나 볼 수 있다.
- 목록에서는 각 게시글 행 오른쪽 위에, 상세에서는 기존 Header 오른쪽 위에 같은 더보기 버튼을 둔다.
- 작성자는 `수정`과 `삭제`를 실행할 수 있다.
- 로그인했지만 작성자가 아닌 사용자가 더보기 버튼을 누르면 `수정 및 삭제는 작성자만 가능합니다.` 안내 팝업과 `확인` 버튼을 표시한다.
- 비로그인 사용자가 더보기 버튼을 누르면 로그인 유도 팝업과 `취소 / 로그인` 버튼을 표시한다.
- 비작성자와 비로그인 사용자에게 수정·삭제 요청을 보내지 않는다.
- 작성자 삭제 시 기존 Hard Delete 확인 팝업과 `복구할 수 없습니다.` 안내, `취소 / 삭제` 버튼을 유지한다.

### 6.2 서버 보안 정책

- UI의 `isOwner`는 표시 제어 전용이다.
- `PATCH /api/posts/:postId`와 `DELETE /api/posts/:postId`는 항상 서버에서 세션, 게시글 존재, `post.user_id === session.user.id`를 다시 검증한다.
- 목록 API는 `isOwner`를 반환할 수 있지만 원본 `user_id`를 공개 응답에 포함하지 않는다.

## 7. 검색 전용 페이지

### 7.1 역할과 라우트

- 검색 아이콘은 `/community/search`로 이동한다.
- 검색 입력은 목록 화면에서 제거하고 검색 전용 페이지에서 제공한다.
- 검색 전용 페이지는 `← 검색` Header를 표시하고 BottomTab을 숨긴다. 뒤로가기는 커뮤니티 목록으로 돌아간다.
- 첫 버전은 제목과 내용 전체를 대상으로 부분 일치·대소문자 무시 검색을 한다.
- 카테고리 검색 필터, 랭킹, 하이라이트, 추천 검색어는 현재 제외한다. API 구조만 확장 가능하게 둔다.

### 7.2 검색 API

- `GET /api/search/posts?q=<keyword>&page=1&limit=10`
- 검색어는 서버에서 trim하고 1~30자로 제한한다.
- 빈 검색어는 검색 결과 API를 호출하지 않는다.
- 초기 정렬은 최신순이며, 검색 정책이 고도화될 때 정렬·랭킹 파라미터를 추가한다.
- 내부적으로 기존 posts 조회 서비스의 검색 로직을 재사용할 수 있으나, 외부 API는 `/api/search/posts`로 분리한다.

### 7.3 최근 검색어

- 검색어는 타이핑 중이 아니라 검색 버튼 또는 Enter로 검색을 실행했을 때만 저장한다.
- 로그인 사용자만 저장한다.
- 비로그인은 검색할 수 있지만 `로그인하면 최근 검색어를 저장할 수 있습니다.` 안내를 본다.
- 사용자별 최대 5개를 최근 실행 순서로 유지한다.
- 같은 검색어를 다시 실행하면 중복 행을 만들지 않고 최신 시각으로 갱신한다.
- 사용자는 자신의 검색어를 하나씩 삭제하거나 전체 삭제할 수 있다.
- 저장과 오래된 기록 정리는 서버에서 호출하는 단일 DB 함수(RPC)로 처리한다. 이 함수는 upsert 후 오래된 기록을 제거하는 하나의 트랜잭션으로 실행해, 동시 요청에도 사용자별 5개 제한을 보장한다.
- 최근 검색어 저장은 검색 결과 API가 성공한 뒤에만 실행한다. 기록 저장에 실패해도 이미 받은 검색 결과는 숨기거나 실패 처리하지 않는다.
- 결과가 0건이어도 검색 API가 성공하면 로그인 사용자의 최근 검색어에 저장한다. 검색 입력만 하고 실행하지 않거나 API 요청이 실패하면 저장하지 않는다.

### 7.4 검색 기록 데이터/API

`search_histories` 테이블을 새로 만든다.

| 필드 | 역할 |
| --- | --- |
| `id` | UUID PK |
| `user_id` | `users.id` FK, `ON DELETE CASCADE` |
| `keyword` | trim·소문자 정규화된 검색어 |
| `updated_at` | 최근 실행 시각 |
| `created_at` | 최초 생성 시각 |

- `(user_id, keyword)`에 unique 제약을 둔다.
- `user_id, updated_at DESC` 인덱스를 둔다.
- 예상 API는 `GET /api/search/histories`, `POST /api/search/histories`, `DELETE /api/search/histories/:historyId`, `DELETE /api/search/histories`다.
- 모든 기록 조회·삭제는 서버에서 현재 세션의 사용자 ID로 범위를 제한한다.
- 기록 생성 API도 trim·소문자 정규화 후 1~30자만 허용한다. 다른 사용자의 기록 ID를 삭제하려는 요청은 존재 여부를 공개하지 않고 실패 처리한다.

## 8. API 응답 경계

DB 필드명과 프론트 응답을 분리한다. DB 변경이 곧바로 UI 계약 변경으로 이어지지 않게 server-side mapper를 둔다. 이는 프론트 변경을 줄이지만, 공개 API 계약 자체를 바꾸는 경우에는 프론트 변경도 필요하다.

목록과 검색 결과의 게시글 DTO는 다음 정보만 제공한다.

```ts
{
  id: string;
  title: string;
  content: string;
  likeCount: number;
  viewCount: number;
  createdAt: string;
  category: { id: string; name: string; slug: string };
  author: { anonymousId: string };
  isOwner: boolean;
}
```

- `user_id`, Google ID, email 등 내부 식별 정보는 공개하지 않는다.
- 기존 성공·실패 API envelope 형식은 유지한다.

## 9. 보류 기능과 미래 확장

### 9.1 배지

- 배지·신뢰도·활동 점수는 현재 구현하지 않는다.
- 배지 테이블, 사용자 배지 연결 테이블, 계산 로직, 화면 pill을 미리 만들지 않는다.
- 실제 사용자 활동 데이터와 운영 정책이 생긴 뒤 별도 설계·검토·구현한다.

### 9.2 검색 고도화

- `/api/search/posts`를 별도 경계로 둔 이유는 이후 검색 랭킹, 제목·내용 하이라이트, 추천 키워드, 카테고리 필터를 독립적으로 추가하기 위해서다.
- 현재는 검색 이벤트를 대량 저장하지 않는다. 최근 검색어 5개만 유지하므로 Supabase 무료 플랜에 부담이 거의 없다.

### 9.3 베스트·인기 콘텐츠

- 현재 좋아요 수와 조회수는 기존 게시글 통계로만 사용한다.
- 베스트 콘텐츠·인기 콘텐츠는 기간, 점수 계산, 노출 위치, 조작 방지 정책을 별도 설계한 뒤 미래 확장으로 구현한다.
- 현재는 관련 API, DB 테이블, 화면을 추가하지 않는다.

### 9.4 이번 확장 범위 밖의 화면

- `/my/posts`는 기존 기능과 접근 권한을 유지한다.
- 내 게시글 목록의 행 디자인, 카테고리 pill, 더보기 메뉴 확장은 이번 범위에 포함하지 않으며 별도 디자인 확장으로 다룬다.

## 10. 구현 순서

1. 카테고리 migration은 `categories` 추가와 nullable `posts.category_id` 추가를 먼저 적용한다.
2. 카테고리 조회 API와 목록·상세·작성·수정 API DTO를 반영한다.
3. 카테고리 필수 선택을 포함한 새 작성·수정 API/UI를 배포한다.
4. `category_id IS NULL` 0건과 FK 상태를 확인한 뒤 `NOT NULL` 제약을 적용한다.
5. 커뮤니티 목록 UI를 행 목록형으로 개편하고 검색 아이콘·필터·카테고리 pill·`+ 글쓰기` pill 적용
6. 목록과 상세의 공통 더보기 메뉴 및 작성자 검증 UX 연결
7. 검색 전용 페이지와 `/api/search/posts` 구현
8. `search_histories` migration, DB RPC, 최근 검색어 API/UI 구현
9. 기존 로그인, 목록, 상세, 작성·수정·삭제, 좋아요, 조회수, 내 정보 회귀 QA

각 단계는 독립 Task로 나누며, SQL 실행·코드 변경·검증이 끝난 뒤 다음 단계로 넘어간다.

### 10.1 Schema Gate

DB 변경이 필요한 Task는 코드 구현 전에 아래 순서를 따른다.

1. Codex가 versioned migration 파일, 실행 전 중단 조건, 실행 후 확인 SQL을 작성한다.
2. 사용자가 SQL 내용을 검토하고 실행을 승인한다.
3. 사용자가 Supabase SQL Editor에서 직접 실행한다.
4. 실행 결과와 확인 SQL 결과를 검토한다.
5. 사용자가 계속 진행을 승인한 뒤에만 Codex가 같은 Task의 API/UI 구현을 진행한다.

- Codex는 Supabase SQL Editor를 직접 실행하지 않는다.
- 예상과 다른 preflight 또는 확인 결과가 나오면 즉시 중단하고 정책을 논의한다.
- 실행된 migration 파일은 수정하지 않는다. 이후 변경은 새 migration 파일로 추가한다.
- SQL이 없는 Task는 Schema Gate 없이 일반 구현·검증 순서를 따른다.

## 11. 완료 검증 기준

- 새 글은 카테고리 없이는 저장되지 않는다.
- 숨김 카테고리 또는 카테고리 관계가 없는 글은 수정 진입 전 재분류 안내를 표시하고, 새 활성 카테고리 선택 전 저장되지 않는다.
- `GET /api/posts`의 slug 기반 필터는 활성 카테고리와 전체 게시글을 정확히 구분하고, 숨김·존재하지 않는 slug는 거부한다.
- 목록과 검색 결과는 익명 ID만 공개하고 내부 사용자 식별자는 노출하지 않는다.
- 작성자만 수정·삭제할 수 있으며, 비작성자 API 직접 호출도 서버에서 거부한다.
- 검색은 비로그인도 가능하고, 최근 검색어 저장·삭제는 로그인 본인에게만 허용된다.
- 검색 기록은 사용자별 다섯 개를 넘지 않고 중복 키워드를 만들지 않는다.
- 동시 검색 요청에도 검색 기록은 사용자별 다섯 개를 넘지 않는다.
- 목록과 상세 모두에서 제목 바로 위 왼쪽 카테고리 pill, 지정된 위치의 더보기 메뉴, `ThumbsUp` 좋아요 수와 `Eye` 조회수가 일관되게 표시된다.
- 목록의 글쓰기 버튼은 `+ 글쓰기` pill로 표시되고, 기존 BottomTab의 구조·경로·활성 상태는 바뀌지 않는다.
- 검색 전용 페이지는 `← 검색` Header를 표시하고 BottomTab을 숨기며, 뒤로가기로 커뮤니티 목록으로 돌아간다.
- `/my/posts`는 이번 확장에서 기존 화면과 동작을 유지한다.
- 좋아요는 `ThumbsUp`, 조회수는 `Eye` 아이콘으로 표시하며, 별도 하트 반응은 만들지 않는다.
- 로그인/로그아웃, Hard Delete와 cascade 동작이 유지된다.
